# Medconsult Test

Тестовый бэкенд на Laravel: API с Sanctum, подтверждение email, профиль, аудит действий (без шума от каждого открытия списка), загрузка и удаление файлов, квота на диск. 

---

## Ответы на вопросы

**Вопрос.** В чём разница между extends и include в Blade?

**Ответ.** extends задаёт наследование от общего макета: страница строится поверх layout, блоки section подставляются в родителе. include просто вставляет фрагмент шаблона (шапка, подвал), без наследования.

**Вопрос.** Что такое сервис-провайдеры и зачем они?

**Ответ.** Классы, которые Laravel поднимает при старте: биндинги в контейнере, маршруты, события, политики. Одна точка настройки до того, как пойдут запросы.

**Вопрос.** Как работает маршрутизация в Laravel?

**Ответ.** Запрос во фронт-контроллер, kernel, роутер сопоставляет метод и URL с правилами из routes. Совпало: middleware по цепочке, потом контроллер или замыкание, ответ клиенту.

**Вопрос.** В чём разница между get и first в Eloquent?

**Ответ.** get возвращает коллекцию всех строк по запросу. first одну первую строку или пусто. Для списка get, для одной записи first.

**Вопрос.** Как сделать миграцию таблицы users с полями имя, email, пароль, статус?

**Ответ.** Миграция через artisan, в up описать таблицу: id, name, уникальный email, password (в БД хранят уже хеш), status, при желании timestamps. Затем migrate.

**Вопрос.** Как получить всех активных пользователей с сортировкой по дате регистрации?

**Ответ.** Фильтр по активному статусу, сортировка по дате регистрации (часто это created_at, если нет отдельного поля).

**Вопрос.** Как связь один ко многим пользователи и посты?

**Ответ.** У постов колонка user_id и внешний ключ. У модели User связь hasMany к постам, у Post belongsTo к пользователю.

## Зачем этот проект

**Версия API:** URL `/api/v1`. Дальнейшие несовместимые изменения: отдельный префикс (например `v2`) или согласованный заголовок `Accept`.

Регистрация с письмом на почту; вход только после подтверждения email. Профиль и файлы только с подтверждённым email. История действий в БД по умолчанию только **audit** (вход, выгрузка, удаление и т.д.). Ошибки API в одном формате: `data`, `message`, `errors`.

Расписание: раз в неделю `user-actions:prune-telemetry` (если включён scheduler). Квота файлов на пользователя: `USER_FILES_QUOTA_BYTES` в `.env`.

## Запуск через Docker (пошагово)

Требуются **Docker** и **Docker Compose** (как правило, поставляются вместе с Docker Desktop или пакетом docker-ce).

### Шаг 1. Каталог проекта

```bash
cd /путь/к/medconsult-test
```

В корне должны находиться `docker-compose.yml` и `Dockerfile`.

### Шаг 2. Сборка образов и запуск контейнеров

```bash
docker compose up -d --build
```

Первый раз будет дольше: собирается PHP 8.3 с расширением pgsql, ставятся зависимости Composer внутри образа. Папка проекта примонтирована в контейнер `app`, код с хоста и в контейнере один и тот же.

### Шаг 3. Подождать старт БД и приложения

В compose два сервиса:

| Сервис | Что делает |
|--------|------------|
| **db** | PostgreSQL 16, база `medconsult`, пользователь `medconsult`, пароль `secret` |
| **app** | после старта `db` выполняет миграции и запускает `php artisan serve` на порту **8089** внутри контейнера |

С хоста БД доступна на **localhost:5433** (в контейнере это порт 5432). Приложение API: **http://localhost:8089**.

Если миграция не применилась из-за готовности БД, однократно выполнить:

```bash
docker compose exec app php artisan migrate --force
```

### Шаг 4. Проверка доступности

Браузер или curl:

- **http://localhost:8089/up** ожидается успешный ответ (health Laravel).
- **http://localhost:8089/** страница welcome.
- Пример API: **http://localhost:8089/api/v1/register** (POST, JSON).

### Шаг 5. Ключ приложения (опционально)

В `docker-compose.yml` для сервиса `app` задан временный `APP_KEY`. Для продакшена или при проблемах с шифрованием ключ генерируется внутри контейнера и прописывается в compose или в `.env`:

```bash
docker compose exec app php artisan key:generate --show
```

### Шаг 6. Остановка

В сессии, где выполнялся `docker compose up`, остановка: **Ctrl+C**. Снятие контейнеров при сохранении данных Postgres в volume `pgdata`:

```bash
docker compose down
```

Снятие контейнеров и удаление тома БД (чистое состояние):

```bash
docker compose down -v
```

### Сводка по окружению

- Базовый URL API: **http://localhost:8089**
- Postgres с хоста: **127.0.0.1:5433**, БД `medconsult`, пользователь `medconsult`, пароль `secret`
- Логи приложения выводятся в консоль той же сессии `docker compose up`

## Документация API

В корне репозитория расположен **`openapi.json`**: в `servers` по умолчанию указан **http://localhost:8089** (Docker). Просмотр через Swagger UI или импорт в Postman.

### Маршруты этого приложения

| Метод | Путь | Доступ | Заметка |
|-------|------|--------|---------|
| GET | `/` | все | Welcome |
| GET | `/up` | все | Проверка, что приложение живое |
| GET | `/email/verify/{id}/{hash}` | signed | Ссылка из письма, без сессии |
| POST | `/api/v1/login` | гость | После подтверждения email |
| POST | `/api/v1/register` | гость | Токен для повторной отправки письма |
| POST | `/api/v1/logout` | Bearer | |
| POST | `/api/v1/email/verification-notification` | Bearer | Повтор письма, пока email не подтверждён |
| GET | `/api/v1/profile` | Bearer + verified | |
| PATCH | `/api/v1/profile` | Bearer + verified | |
| GET | `/api/v1/profile/actions` | Bearer + verified | `per_page`, опционально `include_telemetry=true` |
| GET | `/api/v1/files` | Bearer + verified | `per_page` |
| POST | `/api/v1/files` | Bearer + verified | Multipart `file`, квота |
| GET | `/api/v1/files/{id}` | Bearer + verified | Скачивание |
| DELETE | `/api/v1/files/{id}` | Bearer + verified | Удаление |

## От запроса до ответа (по шагам)

1. Веб-сервер отдаёт запрос в `public/index.php`.
2. Composer, `bootstrap/app.php`, поднимаются провайдеры (register, boot), роуты, контейнер.
3. HTTP kernel, глобальный middleware, роутер ищет маршрут.
4. Middleware маршрута (throttle, sanctum и т.д.), контроллер.
5. Ответ назад через middleware к клиенту.
